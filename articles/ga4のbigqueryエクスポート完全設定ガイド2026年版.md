## 「GA4の管理画面だけでは、もう限界かもしれない…」

ECサイトを運営していて、こんな壁にぶつかっていませんか？

- GA4の探索レポートで14ヶ月以上前のデータが消えていた
- 「購入した人が最初に見たページ」をユーザー単位で追いたいのにできない
- サンプリングがかかって正確な数字が出ない

GA4の管理画面は便利ですが、**生データを自由にSQLで分析できるBigQueryエクスポート**を設定しておくと、これらの課題が一気に解決します。

この記事では、2026年最新のUI・仕様に対応した設定手順を、スクリーンショットなしでも迷わないレベルで解説します。

## 前提条件を確認する

設定を始める前に、以下の3つを確認してください。

| 項目 | 必要な条件 |
|------|-----------|
| GA4プロパティ | 編集者以上の権限を持っていること |
| Google Cloudプロジェクト | 課金アカウントが紐づいていること |
| BigQuery API | 対象プロジェクトで有効化されていること |

:::message
BigQueryには毎月10GBまでの無料ストレージ枠と、毎月1TBまでの無料クエリ枠があります。中小ECサイト（月間数万〜数十万PV程度）であれば、多くの場合無料枠内で運用できます。
:::

## 手順1：Google Cloudプロジェクトを準備する

1. [Google Cloud Console](https://console.cloud.google.com/) にアクセス
2. 左上のプロジェクト選択 → 「新しいプロジェクト」を作成（既存でもOK）
3. 「APIとサービス」→「ライブラリ」→ **BigQuery API** を検索して「有効にする」
4. 「お支払い」からプロジェクトに課金アカウントを紐づける

プロジェクトIDはこの後の設定で使うので、メモしておいてください。

## 手順2：GA4からBigQueryリンクを作成する

1. GA4管理画面 → **管理（歯車アイコン）** → 「プロパティ設定」配下の **「BigQueryのリンク」** を選択
2. 「リンク」をクリック
3. 「BigQueryプロジェクトを選択」で、手順1で準備したプロジェクトを選ぶ
4. データのロケーションを選択（日本なら **`asia-northeast1`（東京）** を推奨）
5. データストリームを選択（通常はすべてチェック）
6. **エクスポートタイプを選択する**

### エクスポートタイプの選び方

| タイプ | テーブル名 | 特徴 | おすすめ用途 |
|--------|-----------|------|-------------|
| 毎日 | `events_YYYYMMDD` | 翌日に確定データが作成される | 日次レポート・定期分析 |
| ストリーミング | `events_intraday_YYYYMMDD` | リアルタイムに近い頻度で更新 | 当日のモニタリング |

:::message alert
ストリーミングエクスポートはBigQueryのストリーミング挿入料金が発生します。中小ECの場合、まずは **「毎日」のみ** で始め、リアルタイム監視が必要になったタイミングでストリーミングを追加するのが安全です。
:::

7. 「送信」をクリックしてリンク完了

設定後、翌日には `analytics_<プロパティID>.events_YYYYMMDD` というテーブルが自動生成されます。

## 手順3：データが入っているか確認する

リンク設定の翌日以降、BigQueryコンソールで以下のSQLを実行してみましょう。

```sql
SELECT
  event_date,
  COUNT(*) AS event_count,
  COUNT(DISTINCT user_pseudo_id) AS unique_users,
  COUNT(DISTINCT
    CONCAT(
      user_pseudo_id,
      '.',
      CAST((SELECT value.int_value FROM UNNEST(event_params) WHERE key = 'ga_session_id') AS STRING)
    )
  ) AS sessions
FROM
  `your-project.analytics_XXXXXXXXX.events_*`
WHERE
  _TABLE_SUFFIX = FORMAT_DATE('%Y%m%d', DATE_SUB(CURRENT_DATE('Asia/Tokyo'), INTERVAL 1 DAY))
GROUP BY
  event_date
ORDER BY
  event_date DESC;
```

:::message
`your-project` と `analytics_XXXXXXXXX` はご自身のプロジェクトIDとGA4プロパティIDに置き換えてください。プロパティIDはGA4管理画面の「プロパティ設定」→「プロパティの詳細」で確認できます。
:::

このクエリで昨日のイベント数・ユニークユーザー数・セッション数が返ってくれば、エクスポートは正常に動作しています。

## 手順4：コストを抑えるための初期設定

BigQueryのコストが心配な方は、以下の2つの設定を追加しておきましょう。

### 古いテーブルの定期削除

GA4のエクスポートテーブルは日付シャーディング（`events_YYYYMMDD`）で日ごとにテーブルが作成されます。保持期限を過ぎたデータを自動削除したい場合は、**Cloud Schedulerと Cloud Functions（またはBigQueryのスケジュールクエリで `DROP TABLE` を実行）で古いテーブルを定期削除するジョブ**を組むのが現実的です。

### カスタムクエリのコスト上限

BigQueryコンソールの「管理」→「プロジェクトの設定」から、**1日あたりのクエリ使用量上限**を設定できます。最初は1TB/日程度に設定しておくと安心です。

## よくあるトラブルと対処法

| 症状 | 原因 | 対処 |
|------|------|------|
| テーブルが作成されない | プロジェクトの課金が無効 | Cloud Consoleで課金アカウントを確認 |
| 権限エラーが出る | サービスアカウントの権限不足 | GA4が使うサービスアカウントにBigQuery編集者ロールを付与 |
| データが少なすぎる | 一部ストリームのみ選択している | BigQueryリンク設定でストリームをすべて選択 |
| `events_intraday` しかない | まだ日次エクスポートが走っていない | 設定翌日の午前中まで待つ |

## まとめ：エクスポート設定はゴールではなくスタート地点

BigQueryエクスポートを設定することで、以下が可能になります。

- **データ保持期限の制約なし**（自分で管理する限り永久保存）
- **サンプリングなしの正確なデータ分析**
- **SQLによる自由度の高いクロス分析**（LTV計算、コホート分析など）

設定自体は30分もあれば完了します。まずはエクスポートを有効にして、生データに触れるところから始めてみてください。
