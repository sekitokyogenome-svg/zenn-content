## はじめに

「GA4は入れてある。BigQueryにもデータが流れている。でも、自分では何も見られない。」

これは1ヶ月前の自分の状態です。EC事業を運営していて、データが大事なのは頭ではわかっている。でも、SQLは書けない。分析ツールの使い方もよくわからない。結局、何か知りたいことがあるたびに詳しい人に聞くか、GA4の標準レポートをなんとなく眺めるだけ。

「今月の売上、先月と比べてどうだっけ？」という単純な疑問すら、自分一人では答えを出せない。

この記事は、エンジニア経験ゼロのEC経営者が**Claude Code × BigQuery MCP**を使って、自分でデータを見られるようになるまでの4週間の記録です。成功したことだけでなく、失敗やつまづきも含めてすべて書きます。

---

## Week 1：Claude CodeとBigQuery MCPのセットアップ

最初の1週間は環境構築に費やしました。正直、ここが一番ハードルが高かった。

### やったこと（ステップバイステップ）

1. **Claude Codeのインストール**
   - Node.jsをインストール（公式サイトからダウンロードするだけ）
   - ターミナルで `npm install -g @anthropic-ai/claude-code` を実行
   - Anthropicのアカウントを作成してAPIキーを取得

2. **BigQuery MCPの設定**
   - GCPプロジェクトでサービスアカウントを作成
   - BigQueryの読み取り権限を付与
   - Claude Codeの設定ファイル（`claude_desktop_config.json`）にMCPサーバーを追加

3. **接続テスト**
   - Claude Codeを起動して「BigQueryのテーブル一覧を見せて」と入力
   - テーブル名が返ってきたときは本当にうれしかった

:::message
BigQuery MCPの設定ファイルは、サービスアカウントのJSONキーのパスを正確に指定する必要があります。パスの区切り文字（Windowsなら`\\`）でつまづく人が多いので注意してください。
:::

### つまづいたポイント

- サービスアカウントの権限設定で「BigQuery閲覧者」だけでは足りず、「BigQueryジョブユーザー」も必要だった
- 設定ファイルのJSON形式を間違えて何度もエラーが出た

## Week 2：はじめての自然言語クエリ

環境ができたら、いよいよデータに触れる番です。

最初に試したのはこんなクエリです。

```
今月の売上を日別で見せて
```

Claude Codeがこの一文からSQLを自動生成し、BigQueryに問い合わせて、結果を表形式で返してくれました。SQLを1行も書いていないのに、日別の売上推移が目の前に表示される。この瞬間、「これはいける」と確信しました。

続けていろいろ聞いてみました。

- 「先週と今週のセッション数を比較して」
- 「デバイス別のコンバージョン率を出して」
- 「一番売れている商品カテゴリはどれ？」

どれも数秒〜十数秒で回答が返ってきます。

:::message alert
自然言語クエリは便利ですが、最初のうちはClaude Codeが生成したSQLも一緒に確認することをおすすめします。意図しないテーブルやカラムを参照していることがたまにあります。
:::

## Week 3：もう少し深い分析に挑戦

基本的なクエリに慣れたら、次はもう少し踏み込んだ分析をやってみました。

### チャネル別の比較

```
チャネル別に売上とセッション数を比較して、ROASが高い順に並べて
```

Organic Search、Direct、Paid Search、Socialなど、チャネルごとのパフォーマンスが一覧で出てきます。これまでGA4の管理画面を行ったり来たりしていた作業が、一発で完了します。

### ファネル分析

```
先月の商品閲覧→カート追加→購入のファネルを見せて。各ステップの転換率も出して
```

ファネルの数字を見たのは正直これが初めてでした。カート追加から購入への転換率が思っていたよりずっと低いことがわかり、カートページの改善に着手するきっかけになりました。

### コホート分析

```
初回訪問月ごとに、翌月以降のリピート率を出して
```

コホート分析という言葉自体、この時期に初めて知りました。Claude Codeに「コホート分析って何？」と聞いたら、概念の説明と一緒にデータも出してくれた。学びながら分析できるのがありがたかったです。

## Week 4：日常のルーティンに組み込む

4週目に入ると、分析がだんだん日常業務の一部になってきました。

### 毎朝のチェッククエリ

毎朝、以下の3つを確認するのが習慣になりました。

1. 「昨日の売上とセッション数を教えて」
2. 「直近7日間で異常値がないかチェックして」
3. 「昨日のチャネル別パフォーマンスを見せて」

所要時間は合計で2〜3分です。以前はGA4を開いてもどこを見ればいいかわからず、結局何も確認しないまま1日が過ぎていました。

### 週次レポートの作成

金曜日には少し時間をかけて、週次の振り返りをClaude Codeに依頼しています。

```
今週と先週を比較して、売上・セッション・CVRの変化をまとめて。
特に大きく変動した項目があれば理由の仮説も出して。
```

これをそのままメモに貼り付けて、翌週の施策検討に使っています。

## 失敗と学びの記録

4週間を通じて、いくつもの失敗がありました。同じことで困る人がいるかもしれないので、正直に書いておきます。

### テーブル名の間違い

GA4のBigQueryエクスポートは `events_YYYYMMDD` という日付サフィックスのテーブル構成です。最初はこの仕組みを知らず、「テーブルが見つからない」というエラーに何度も遭遇しました。Claude Codeにテーブル構造を説明してから聞くようにしたら、この問題は解消されました。

### 指標の誤解

「セッション数」と「ユーザー数」の違いを正しく理解していなかったため、分析結果を間違って解釈していた時期がありました。わからない指標が出てきたら、その場でClaude Codeに定義を聞くクセをつけるのが大事です。

### 期間指定のミス

「今月の売上」と言ったつもりが、Claude Codeが参照した期間がずれていたことがありました。曖昧な表現よりも「2026年3月1日から3月28日まで」のように明示するほうが正確です。

---

## 変わったこと：「誰かに聞く」から「自分で30秒で確認する」へ

この4週間で一番変わったのは、**データに対する距離感**です。

以前は「売上が下がっている気がする」と思っても、確認する手段がなく、不安を抱えたまま過ごしていました。今は気になったらすぐClaude Codeに聞ける。答えが30秒で返ってくる。

これは単なる効率化ではなく、**意思決定のスピードと質が根本的に変わる**体験でした。

たとえば、ある広告キャンペーンの効果を翌日には数字で確認できるようになった。以前なら「たぶん効いている気がする」で1週間放置していたものが、翌朝には「セッションは増えたけどCVRが下がっている。LPを見直そう」と判断できる。

## 非エンジニアがClaude Codeを始めるためのヒント

同じような立場の方に向けて、いくつかアドバイスを書いておきます。

1. **最初の環境構築は誰かに手伝ってもらっていい** — セットアップだけは技術的な知識が要る部分があります。ここだけ詳しい人に頼んでも、その後は自走できます。

2. **簡単な質問から始める** — 「昨日の売上は？」くらいの質問から始めて、少しずつ複雑にしていくのがコツです。

3. **生成されたSQLも見る習慣をつける** — 理解する必要はありませんが、眺めるだけでも「何をやっているか」がだんだんわかってきます。

4. **データの定義を確認する** — 「売上」が税込なのか税抜なのか、「セッション」の定義は何か。曖昧なまま進めると、後で混乱します。

5. **間違いを恐れない** — 読み取り専用で接続していれば、どんなクエリを投げてもデータが壊れることはありません。

---

## それでもプロの力が必要な場面

Claude Codeで自走できるようになったとはいえ、すべてを一人で完結できるわけではありません。

- **初期のデータ基盤設計**（BigQueryのテーブル構成、stagingやmartの設計）
- **GA4の計測タグ設定**（そもそも正しいデータが取れていなければ分析の意味がない）
- **複雑なデータモデリング**（コホート分析やLTV計算の設計）
- **Looker Studioでのダッシュボード構築**（チーム全体で共有する仕組み）

こうした部分は、一度プロに相談して正しい基盤を作ってもらうほうが、長い目で見て効率的です。基盤さえ整えば、日常の分析はClaude Codeで自走できます。

## まとめ

非エンジニアのEC経営者でも、Claude Code × BigQuery MCPを使えば自分でデータを見る習慣を作れます。

- セットアップは1週間。つまづくポイントはあるが乗り越えられる
- 自然言語で聞くだけで、日別売上からファネル分析まで対応できる
- 毎朝のチェックを習慣にすると、意思決定のスピードが変わる
- 失敗しながら覚えるのが一番の近道
- データ基盤の構築や計測設計など、プロに任せるべき部分もある

「データは溜まっているのに活用できていない」という状態は、もったいないだけでなく、経営判断の機会損失でもあります。
