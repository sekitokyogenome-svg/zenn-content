## 「GA4の管理画面だけでは限界がある…」と感じていませんか？

「GA4の探索レポートで90日以上前のデータが消えていた」「セグメントを細かく掛けると(other)にまとめられてしまう」——EC運営をしていると、こうした壁にぶつかる瞬間があります。

その解決策が **GA4 → BigQueryエクスポート** です。生データをBigQueryに蓄積すれば、データ保持期間の制限もサンプリングも関係なくなります。

本記事では、2026年最新のUI・仕様に対応した設定手順を、スクリーンショットの代わりにステップごとの操作パスで丁寧に解説します。

---

## 前提条件の確認

設定を始める前に、以下の3点を確認してください。

| 項目 | 必要な状態 |
|---|---|
| GA4プロパティ | 編集者以上の権限 |
| Google Cloudプロジェクト | オーナーまたは編集者の権限 |
| BigQuery API | 対象プロジェクトで有効化済み |

:::message
Google Cloudプロジェクトがまだない場合は、[Google Cloud Console](https://console.cloud.google.com/) で無料で作成できます。BigQueryにはサンドボックス（無料枠）がありますが、エクスポート先としては課金アカウントの紐づけが必要です。
:::

## ステップ1：Google Cloud側の準備

### 1-1. BigQuery APIの有効化

```
Google Cloud Console
→ APIとサービス → ライブラリ
→ 「BigQuery API」を検索 → 有効にする
```

### 1-2. データセットのロケーション確認

エクスポート時にデータセットは自動作成されますが、**ロケーション（リージョン）は後から変更できません**。日本のECサイトであれば `asia-northeast1`（東京）を推奨します。

:::message alert
既存データセットを指定する場合、GA4側で選択したロケーションとデータセットのロケーションが一致しないとエラーになります。事前に確認しましょう。
:::

---

## ステップ2：GA4管理画面でのエクスポート設定

```
GA4 管理画面
→ プロパティ列 → プロダクトリンク → BigQueryのリンク
→ 「リンク」をクリック
```

設定画面で入力する項目は以下の通りです。

| 設定項目 | 推奨値 | 補足 |
|---|---|---|
| Google Cloudプロジェクト | 自社のプロジェクトを選択 | |
| データのロケーション | asia-northeast1（東京） | 後から変更不可 |
| データストリーム | 対象のウェブストリームを選択 | 複数選択可 |
| 頻度 | **毎日＋ストリーミング** | 下記で詳しく解説 |
| エクスポートするイベントデータ | すべてのイベント | フィルタも可能 |

### 「毎日」と「ストリーミング」の違い

- **毎日（Daily）**: 日次バッチで `events_YYYYMMDD` テーブルが作成される。**無料**。
- **ストリーミング（Streaming）**: リアルタイムに `events_intraday_YYYYMMDD` テーブルへ書き込まれる。BigQueryのストリーミング挿入料金が発生。

:::message
まずは **毎日エクスポートのみ** で始めるのがおすすめです。月間100万PV規模のECサイトでも、日次テーブルのストレージ費用は月額数十円〜数百円程度で収まるケースがほとんどです。リアルタイム分析が必要になった段階でストリーミングを追加しましょう。
:::

設定が完了すると、**翌日からデータが蓄積**され始めます。過去データの遡り取得はできないため、設定は早ければ早いほど有利です。

## ステップ3：データが入っているか確認する

エクスポート翌日以降、以下のSQLをBigQueryで実行して確認します。

```sql
-- エクスポートされたイベント数とセッション数を確認
SELECT
  event_date,
  COUNT(*) AS event_count,
  COUNT(DISTINCT
    CONCAT(
      user_pseudo_id,
      CAST(
        (SELECT value.int_value FROM UNNEST(event_params) WHERE key = 'ga_session_id')
        AS STRING
      )
    )
  ) AS session_count
FROM
  `your-project.analytics_XXXXXXXXX.events_*`
WHERE
  _TABLE_SUFFIX = FORMAT_DATE('%Y%m%d', DATE_SUB(CURRENT_DATE('Asia/Tokyo'), INTERVAL 1 DAY))
GROUP BY
  event_date
ORDER BY
  event_date;
```

:::message
`your-project` と `analytics_XXXXXXXXX` は実際のプロジェクトIDとデータセット名に置き換えてください。データセット名の数字部分はGA4プロパティIDです。
:::

結果に `event_count` が表示されれば、エクスポートは正常に動作しています。

## よくあるトラブルと対処法

### テーブルが作成されない

- **原因1**: Google Cloudプロジェクトに課金アカウントが紐づいていない
- **原因2**: GA4側のリンク設定でストリームを選択し忘れている
- **対処**: GA4管理画面の「BigQueryのリンク」から設定内容を再確認

### データが途中で欠落する

- **原因**: `events_intraday_*` テーブルは日次確定テーブル（`events_YYYYMMDD`）が作成されると自動削除されます。中間テーブルの消失はエラーではありません。

### クエリ実行時に課金が心配

BigQueryはオンデマンド料金の場合、**スキャン量**に応じた課金です。`_TABLE_SUFFIX` で日付を絞ったり、必要なカラムだけSELECTすることでコストを抑えられます。毎月1TBまでは無料枠があるため、小〜中規模ECサイトなら無料枠内で収まることも多いです。

## まとめ：設定は「今日」やっておくべき

- GA4 → BigQueryエクスポートは **過去に遡れない**
- 日次エクスポートだけなら **ストレージ費用はごくわずか**
- 一度設定すれば、あとは自動で生データが蓄積され続ける

「いつかやろう」と思っている間にも、分析に使えたはずのデータは消えていきます。設定自体は10分で完了するので、この記事を読んだ今、手を動かすのがベストなタイミングです。
