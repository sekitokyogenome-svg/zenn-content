## 何を作ったか

アンケートモニター（座談会・会場調査・Webアンケート）の案件は、メールで届く。ここに2つの困りごとがあった。

1. **量が多い**。十数サイトに登録していると、1日に何十通も届く。
2. **おいしい案件ほど早く埋まる**。会場調査やインタビューは謝礼5,000〜15,000円と高単価だが、たいてい「先着」「本日中」。気づいたときには締切済み。

そこで、Gmailに届くアンケートメールを**自動でパースして、謝礼額・締切・種別でスコアリングし、価値が高いものだけLINEに通知する**システムを作った。名付けて Survey Catcher。

この記事では設計判断と実装のキモ——特に**表記ゆれに強い正規表現パース**と**誤検出の潰し方**、そして**プライバシー設計**——を、実際のコードを引きながら書く。

## 全体構成

```
Cloud Scheduler（15分毎）
        │ POST /tasks/run
        ▼
   Cloud Run（FastAPI）
        │
        ├─ Gmail API（readonly）でメール取得
        ├─ パース：謝礼 / 締切 / 種別 を抽出
        ├─ スコアリング：謝礼 × 締切の近さ × 種別重み
        ├─ Firestore に保存（本文は破棄）
        └─ 高スコアだけ LINE Messaging API で push
```

ポイントは、**各サイトへの自動ログインやスクレイピングを一切しない**こと。サービスごとの規約・BANリスクを避け、自分宛てに届いたメールだけを正規の Gmail API で読む。入力を「自分のメール」に限定することで、グレーな領域に踏み込まずに済む。

## 設計判断1：メール本文は保存しない

最初に決めたのは「**本文を永続化しない**」こと。

アンケートメールには回答用のワンタイムURLや、個人に紐づく情報が載ることがある。これをDBに溜め込むのは規約的にもプライバシー的にも避けたい。

そこで、パースに使うのは取得直後のメモリ上の本文だけにして、Firestoreに残すのは**抽出済みメタデータ＋先頭200字程度の抜粋**に限定した。

```python
class Case(BaseModel):
    """案件（Firestore の cases コレクション 1 ドキュメント）。"""
    gmail_msg_id: str
    source: str            # どのアンケートサイトか
    title: str
    received_at: datetime
    excerpt: str = ""      # 本文先頭〜200字。全文は破棄する。
    url: Optional[str] = None

    case_type: CaseType = CaseType.UNKNOWN
    reward_yen: Optional[int] = None
    is_first_come: bool = False
    deadline_at: Optional[datetime] = None
    score: float = 0.0
    # ...
```

OAuthスコープも `gmail.readonly` だけにした。送信も変更もできないトークンしか持たない。個人開発でも、扱うデータの最小化を最初に設計で縛っておくと後がラクだ。

## 設計判断2：パースは正規表現が主役、LLMは補助

「メールのパースなんて全部LLMに投げればいいのでは？」とも考えた。が、

- 15分毎に十数〜数十通を回すと、**LLM呼び出しのコストと遅延**が無視できない
- 謝礼額や日付の抽出は**ルールが明確で、正規表現の方が速くて正確**

という理由で、**抽出は正規表現を主役**にした。LLMは月予算ガードを付けたうえで、別機能（朝のブリーフィング要約や記事生成）に回している。「LLMでできる」と「LLMでやるべき」は別、という判断だ。

ここからが本題。日本語のアンケートメールは**表記ゆれが激しい**ので、その吸収が実装の山場になる。

## 実装のキモ1：謝礼額の抽出

謝礼の書き方は本当にバラバラだ。

- `8,000円` / `5,000pt` / `500ポイント`
- `1.5万円` / `5〜15万円`（レンジ）
- `Amazonギフト 1,000円分`

まず単位ごとにパターンを用意する。地味にハマったのが**ポイント表記**で、`500pt進呈` のように直後が漢字だと `\b`（単語境界）が効かない。日本語と英数字の境目では `\b` が成立しないからだ。そこで「直後が英字でないこと」を否定先読みで書いた。

```python
_DASH = r"[〜～~ー－—–\-]"
# 「5〜15万円」レンジ（最大値=後者を採用）
_MAN_RANGE_RE = re.compile(rf"([\d,]+(?:\.\d+)?)\s*{_DASH}\s*([\d,]+(?:\.\d+)?)\s*万円")
# 「8,000円」
_YEN_RE = re.compile(r"([\d,]+)\s*円")
# 「500pt」「500ポイント」。\b が和文前で効かないため否定先読みを使う
_PT_RE = re.compile(r"([\d,]+)\s*(?:pts|pt|ポイント分|ポイント|p)(?![A-Za-z])", re.IGNORECASE)
```

ところが実メールで動かすと、**誤検出**が大量に出た。広告文に紛れる金額を謝礼と勘違いするのだ。

- 「**年収**500万円の人向け調査」→ `5,000,000円` を謝礼と誤認
- 「総額100万円分の**懸賞**！」→ 懸賞の総額を拾う
- 「最新ガジェットが**当たる**」系の煽り文句

対策を2段で入れた。

**(a) サニティ上限**：そもそも謝礼が50万円を超えることはまずないので、それを超える抽出値は捨てる。

**(b) 近接文脈フィルタ**：金額の前後（±25文字）に「謝礼・報酬・進呈・ポイント」等があれば優先採用し、「年収・懸賞・投資・抽選・相当」等があれば除外する。

```python
ctx_pos  = [m.start() for kw in context_keywords for m in re.finditer(re.escape(kw), text)]
excl_pos = [m.start() for kw in exclude_context  for m in re.finditer(re.escape(kw), text)]

def near(positions, c):
    s, e = c[0], c[1]
    return any(min(abs(s - p), abs(e - p)) <= _CONTEXT_WINDOW for p in positions)

# 「謝礼語の近く」を最優先。無ければ「非謝礼語の近く」を除外した候補から選ぶ
ctx_cands = [c for c in candidates if near(ctx_pos, c)]
pool = ctx_cands if ctx_cands else [c for c in candidates if not near(excl_pos, c)]
best = max(pool, key=lambda c: c[2])  # その中の最大額を採用
```

この「**近接文脈でフィルタする**」発想は、周辺ノイズの多い日本語テキストの固有値抽出に広く効く。キーワードはYAMLに外出しして、実データを見ながら育てている。正規表現は「書いて終わり」ではなく、誤検出を観測してキーワードを足していく運用が本番だ。

## 実装のキモ2：締切の抽出

締切も絶対表記と相対表記が混在する。

- 絶対：`6月30日`、`6/30 23:59`
- 相対：`本日中`、`明日まで`、`あと3時間`
- そもそも日時がなく `先着` だけ

相対表記を先に判定し（最も具体的なので）、無ければ絶対表記の候補を集める。絶対表記では「複数の日付のうちどれが締切か」が問題になるので、**`締切`・`まで`・`応募` 等が近い日付を優先**し、その中で受信日以降の最も早い日時を採用する。

```python
# 受信時点より過去の月日なら翌年とみなす（年またぎ対策）
def _build(received, month, day, hour, minute):
    year = received.year
    if (month, day) < (received.month, received.day):
        year += 1
    return datetime(year, month, day, hour, minute, tzinfo=JST)
```

`12月30日` 受信のメールに `1月5日締切` とあれば翌年、という年またぎ補正は、実運用だと地味に効く。

## 実装のキモ3：スコアリング

抽出できたら、案件の「いま追うべき度」を0〜100で点数化する。式はシンプルに**3要素の重み付き和**。

```
score = 0.40 × 謝礼の正規化値
      + 0.35 × 締切の近さ
      + 0.25 × 種別重み
```

締切の近さは、締切まで `h` 時間として `1 / (1 + h/24)` の逓減カーブにした。締切0時間で1.0、24時間で約0.5、72時間で約0.25と、近いほど急に効く。

```python
def _urgency(deadline_at, is_first_come, now, cfg):
    if deadline_at is None:
        return cfg["first_come_urgency"] if is_first_come else cfg["no_deadline_urgency"]
    hours = (deadline_at - now).total_seconds() / 3600.0
    if hours <= 0:
        return 0.0  # 締切超過は緊急度ゼロ
    return min(1.0 / (1.0 + hours / 24.0), 1.0)
```

種別重みは、**会場調査・インタビューを最優先（1.0）**、本調査0.8、Webアンケート0.3。高単価で枠が少ない対面型を上に持ってくる設計だ。この重みもYAMLで調整できる。

## 通知：鳴りすぎないことが正義

最初に作ったとき、通知が鳴りすぎて失敗した。深夜にも鳴るし、無料枠（LINE Messaging APIのpushは月200通まで）もすぐ尽きる。

そこで通知側にいくつもガードを足した。

- **スコア閾値**：一定点未満は保存はするが通知しない
- **謝礼下限**：謝礼10円のような低単価は、締切間近でも鳴らさない
- **ダイジェスト**：15分毎の実行で複数件出ても、まとめて1通にする
- **静音時間**：22〜翌7時は通知せず、翌朝にまとめる
- **月間上限**：push回数に上限を設け、無料枠を超えない

「価値が高いものだけ、起きている時間に、まとめて」。通知設計は機能そのものと同じくらい体験を左右した。

## インフラ：個人開発のサーバーレス構成

- **Cloud Run**：FastAPIアプリをコンテナで。アイドル時はゼロスケールで課金されない
- **Cloud Scheduler**：15分毎に `/tasks/run` を叩くだけ。エンドポイントはトークンヘッダで保護
- **Firestore**：案件・通知状態を保存。サーバーレスで無料枠が広い
- **Secret Manager**：OAuthトークンやAPIキーを注入

定期実行系の個人開発は、この「Scheduler → Cloud Run → Firestore」の3点セットが鉄板。常時起動のVMを持たずに済むので、月数十円〜のレンジで回る。

## 実際に運用してみて

数週間動かしてわかったのは、**サイトによって案件の質がまるで違う**ことだ。Webアンケート中心で単価の低いサイトもあれば、会場調査やインタビューの高単価案件がコンスタントに届くサイトもある。Survey Catcherが貯めたメタデータ（謝礼額・種別・頻度）を集計すると、自分にとって「通知に値するサイト」が定量的に見えてくる。

ちなみに監視対象の一例が [infoQ](https://infoq.jp/referral/index/hpz6dsycqK68/04)（GMOリサーチ運営）。Web案件から座談会まで表現のバリエーションが豊富に届くので、**パーサの表記ゆれ対策のテストデータ供給源**として一番重宝した。同種の仕組みを作るなら、登録が無料でメール量の多いサイトを1つ繋いでおくと、実データで正規表現を鍛えられて開発が捗る（※このリンクは紹介リンクを含むPRです）。

## まとめ

- アンケート案件メールを**正規表現主体でパース**し、謝礼×締切×種別でスコアリングして、価値の高いものだけLINE通知する仕組みを作った
- 日本語テキストの抽出は、**表記ゆれの吸収**と**近接文脈による誤検出フィルタ**が肝
- 扱うデータは最小化（**本文は破棄・readonlyスコープ**）、サイトへの自動ログインはしない
- 通知は「鳴らさない工夫」が機能の質を決める
- インフラは Scheduler → Cloud Run → Firestore のサーバーレス3点セットで安く回る

正規表現とちょっとしたスコアリングだけでも、日々の情報収集はかなり自動化できる。同じように「メールで届く何か」を追っている人の参考になれば。

## コードと、ひとつ質問

本文に載せた謝礼・締切のパーサとスコアリングのロジックは、これだけでもそのまま実用になる。抜き出して自分のユースケース（請求書メール、予約確認、通知の振り分けなど）に流用できるはずだ。全体のコードは整理中で、**需要がありそうなら一式公開しようと思う——読みたい人はコメントやいいねで反応してもらえると助かる**。

最後に逆に聞きたい。**表記ゆれに強いテキスト抽出、みなさんはどうしていますか？** 正規表現を泥臭く育てる派か、最初からLLMに投げる派か、ハイブリッドか。「メールで届く何か」を自動処理している人の工夫を、コメントで教えてもらえると嬉しいです。
