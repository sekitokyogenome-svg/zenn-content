## GTMの設定変更が「誰が何を変えたかわからない」状態になっていませんか

GTMを複数人で運用していると、以下のような問題が起きがちです。

- 誰かが変更した設定が意図せずリセットされる
- テスト中のタグが本番に公開されてしまう
- 変更の経緯がわからず、元に戻せない

GTMにはGitのようなブランチ機能に相当する「ワークスペース」と、デプロイ履歴を管理する「バージョン」の仕組みがあります。
これらを正しく運用すれば、チーム開発でも安全にタグ管理を行えます。

この記事では、GTMのワークスペースとバージョン管理の運用ルールを解説します。

## ワークスペースの基本概念

GTMのワークスペースは、Git のブランチに近い概念です。

### デフォルトワークスペース

GTMコンテナには「Default Workspace」が1つ存在します。
これがメインの作業領域で、ここから新しいワークスペースを作成できます。

### ワークスペースの上限

| GTMプラン | ワークスペース数 |
|-----------|----------------|
| 無料版 | 3つまで |
| 360（有料版） | 無制限 |

無料版では3つという制約があるため、ワークスペースの運用ルールを決めておく必要があります。

### ワークスペースの推奨運用パターン

```
[Default Workspace]  → 本番環境の現在の状態（直接編集しない）
[Dev - 担当者名]     → 開発・テスト用
[Hotfix]             → 緊急修正用（常に空けておく）
```

無料版の3枠を上記のように割り振ると、通常の開発と緊急対応を両立できます。

## ワークスペースの作成と運用フロー

### 手順1：新しいワークスペースを作成

1. GTM管理画面の左上にある現在のワークスペース名をクリック
2. 「+」ボタンで新しいワークスペースを作成
3. 名前と説明を入力

**命名規則の例：**

```
[目的] - [担当者] - [日付]
例: AddScrollTracking - Tanaka - 20260330
```

### 手順2：ワークスペースで作業する

作成したワークスペース内でタグ・トリガー・変数を編集します。
この変更は他のワークスペースには影響しません。

### 手順3：プレビューモードでテスト

ワークスペース単位でプレビューモードが使えます。
本番環境に影響を与えず、変更内容をテストできます。

### 手順4：送信（公開）

テストが完了したら「送信」ボタンでバージョンを作成し、本番に反映します。

## バージョン管理の仕組み

GTMの「バージョン」は、コンテナの特定時点のスナップショットです。

### バージョンの構成

| 要素 | 内容 |
|------|------|
| バージョン番号 | 自動採番（1, 2, 3...） |
| バージョン名 | 任意の名前（手動入力） |
| 説明 | 変更内容の説明（手動入力） |
| 作成者 | 送信したユーザー |
| 作成日時 | タイムスタンプ |

### バージョン名と説明の記述ルール

バージョン名と説明は、将来の自分やチームメンバーが変更内容を理解するために重要です。

**バージョン名のフォーマット例：**

```
v[番号] [変更概要]
例: v15 スクロール深度トリガー追加
例: v16 LINE Tag コンバージョン修正
```

**説明のフォーマット例：**

```
## 変更内容
- GA4イベントタグ「scroll_depth」を新規作成
- スクロール深度トリガー（25/50/75/100%）を新規作成
- 組み込み変数「Scroll Depth Threshold」を有効化

## テスト結果
- プレビューモードでトリガー発火を確認済み
- GA4 DebugViewでイベント受信を確認済み

## 担当
田中
```

## コンフリクト（競合）の回避

### コンフリクトが発生する条件

2つのワークスペースで同じタグ・トリガー・変数を編集し、片方を先に公開した場合、もう片方のワークスペースでコンフリクトが発生します。

### コンフリクトの検出

ワークスペースの「変更の概要」画面に、コンフリクトの警告が表示されます。
GTMは自動マージを行わないため、手動で解決する必要があります。

### コンフリクト解決の手順

1. コンフリクトが発生しているワークスペースを開く
2. 「変更を同期」ボタンをクリック
3. 各コンフリクト項目について「自分の変更を使用」または「公開中のバージョンを使用」を選択
4. 同期完了後、プレビューモードで再テスト

### コンフリクトを防ぐ運用ルール

```
1. 作業前にSlackで「GTMの〇〇を編集します」と宣言する
2. 同じタグ・トリガーの同時編集を避ける
3. ワークスペースは作業完了後すぐに公開して閉じる
4. 長期間ワークスペースを開いたまま放置しない
```

## 承認フローの構築

チーム開発では、誰でも自由に本番公開できる状態はリスクがあります。
GTMの権限設定を使って承認フローを構築します。

### GTMのユーザー権限レベル

| 権限レベル | できること |
|-----------|-----------|
| 読み取り | コンテナの閲覧のみ |
| 編集 | タグ・トリガー・変数の編集 |
| 承認 | バージョンの作成（公開はできない） |
| 公開 | バージョンの作成と本番公開 |

### 推奨する権限設計

| 役割 | 権限レベル |
|------|-----------|
| マーケティング担当者 | 編集 |
| GTM管理者 | 承認 |
| リーダー / 責任者 | 公開 |

この構成にすると、以下のフローになります。

```
[マーケ担当] ワークスペースでタグを編集
  ↓
[マーケ担当] プレビューモードでテスト
  ↓
[マーケ担当] 「レビュー依頼」として管理者に連絡
  ↓
[GTM管理者] 変更内容を確認し、バージョンを作成
  ↓
[リーダー] 最終確認して本番公開
```

### 権限の設定方法

1. GTM管理画面の「管理」→「ユーザー管理」
2. ユーザーを追加し、コンテナ権限を設定
3. 「コンテナの権限」で適切なレベルを選択

## ロールバック（巻き戻し）の手順

公開後に問題が発生した場合、以前のバージョンに戻せます。

### 手順

1. GTM管理画面の「バージョン」タブを開く
2. 戻したいバージョンの「アクション」→「公開」をクリック
3. 確認画面で「公開」を実行

:::message
ロールバックすると新しいバージョン番号が採番されます。
バージョン履歴は保持されるため、ロールバック前の状態にも再度戻すことが可能です。
:::

## チーム運用のベストプラクティス

### 命名規則の統一

GTMコンテナ内の要素すべてに接頭辞を付けると、管理が格段に楽になります。

| 要素 | 接頭辞 | 例 |
|------|--------|-----|
| タグ | GA4 Event - / LINE - / TikTok - | GA4 Event - scroll_depth |
| トリガー | CE - / PV - / Click - | CE - purchase |
| 変数 | DLV - / URL - / CJS - | DLV - item_name |
| フォルダ | （カテゴリ名） | GA4 / 広告タグ / ユーティリティ |

### フォルダによる整理

GTMにはフォルダ機能があります。
タグ・トリガー・変数をカテゴリごとにフォルダ分けすると、数十個以上の要素があっても見通しが良くなります。

**推奨フォルダ構成：**

```
GA4/
  GA4 Config
  GA4 Event - purchase
  GA4 Event - scroll_depth
広告タグ/
  LINE Tag - Base Code
  LINE Tag - Conversion
  TikTok Pixel - Base Code
  TikTok Pixel - CompletePayment
ユーティリティ/
  Cookie同意バナー連携
  デバッグ用タグ
```

### 定期的なコンテナの棚卸し

月に1回程度、以下の項目をチェックします。

- 使われていないタグ（一時停止中のタグ）がないか
- テスト用に作ったワークスペースが残っていないか
- 不要になった変数やトリガーがないか

GTMの「管理」→「コンテナの変更」から、最近の変更履歴を確認できます。

## GTM APIを使った変更履歴の自動取得

GTMにはAPIがあり、バージョン情報やコンテナの変更をプログラムで取得できます。

```javascript
// Node.jsでGTM APIからバージョン一覧を取得する例
const { google } = require('googleapis');

async function listVersions() {
  const auth = new google.auth.GoogleAuth({
    scopes: ['https://www.googleapis.com/auth/tagmanager.readonly']
  });

  const tagmanager = google.tagmanager({ version: 'v2', auth });

  const res = await tagmanager.accounts.containers.version_headers.list({
    parent: 'accounts/ACCOUNT_ID/containers/CONTAINER_ID'
  });

  res.data.containerVersionHeader.forEach(version => {
    console.log(`v${version.containerVersionId}: ${version.name}`);
    console.log(`  作成者: ${version.email}`);
    console.log(`  日時: ${version.numMacros}`);
  });
}

listVersions();
```

この仕組みを使えば、変更履歴をSlackに自動通知する連携も構築できます。

## まとめ

GTMのワークスペースとバージョン管理のポイントを整理します。

- ワークスペースはGitのブランチのように使い、直接Default Workspaceを編集しない
- 無料版は3つまでの制約があるため、用途を明確にして運用する
- バージョン名と説明を丁寧に記述し、変更の経緯を残す
- コンフリクトは作業前の宣言と短期間での公開で防ぐ
- 権限設計で承認フローを構築し、意図しない公開を防止する
- 命名規則とフォルダ整理で大規模なコンテナも管理しやすくなる

GTMの運用ルールを整備すると、チームの生産性とデータ品質の両方が向上します。
