「GA4をBigQueryにつないでみたけど、データの見方がわからない」――これは、GA4×BigQueryを導入した直後のEC事業者が高確率で直面する感想です。

GA4のBigQueryエクスポートを設定するのは、実はそこまで難しくありません。GCPプロジェクトを作成し、GA4の管理画面でエクスポートを有効にするだけです。しかし、その後が問題です。

この記事では、某EC事業者がGA4×BigQueryを自社導入してからの最初の1週間を、体験記風に振り返ります。導入直後のつまずきポイントと、そこから得られたインサイトを共有します。

## 導入前日：期待と不安

GA4の管理画面でBigQueryリンクを設定しました。「ストリーミング」と「日次」の2つのエクスポート方式があり、まずは日次エクスポートを選択。ストリーミングは費用が発生するため、最初は日次で様子を見ることにしました。

設定自体は10分程度で完了。「明日からデータが溜まり始めるはず」と思い、その日は終了しました。

## Day 1：テーブルが見つからない

翌朝、BigQueryのコンソールを開くと、データセットは作成されているものの、テーブルが見当たりません。

少し焦りましたが、調べてみると、日次エクスポートは翌日のデータから反映されるため、設定当日のデータは含まれないことがわかりました。さらに、エクスポートのタイミングは設定から24〜48時間後になることもあります。

**気づき：データが反映されるまでには時間がかかる。焦らず待つ。**

## Day 2：テーブル構造に驚く

2日目、ついに `events_YYYYMMDD` というテーブルが作成されていました。さっそく中身を見てみると、想像していたものとまったく違いました。

GA4の標準レポートは「ページビュー」「セッション数」「ユーザー数」など、わかりやすい指標で表示されます。しかし、BigQueryにエクスポートされるデータは、すべてが「イベント」単位のフラットなデータです。

```sql
-- テーブルの中身を確認
SELECT event_name, COUNT(*) as event_count
FROM `project.analytics_XXXXXXX.events_20260329`
GROUP BY event_name
ORDER BY event_count DESC
LIMIT 20
```

`page_view`、`session_start`、`user_engagement`、`scroll`……。イベント名が並んでいますが、ページURLやセッションIDはどこにあるのか。

答えは `event_params` というネストされたカラムの中でした。GA4のBigQueryデータでは、イベントのパラメータがRECORD型のネストされた構造で格納されています。

**気づき：GA4のBigQueryデータは、そのままでは分析しにくい。UNNESTを使ったフラット化が必要。**

## Day 3：UNNESTとの格闘

3日目は、`event_params` からページURLを取り出す作業に費やしました。

```sql
-- event_paramsからpage_locationを取得
SELECT
  event_timestamp,
  event_name,
  (SELECT value.string_value FROM UNNEST(event_params) WHERE key = 'page_location') AS page_location
FROM `project.analytics_XXXXXXX.events_20260329`
WHERE event_name = 'page_view'
LIMIT 100
```

この `UNNEST` の書き方に慣れるまでに時間がかかりました。GA4のBigQueryデータを扱うには、`UNNEST` が避けて通れないスキルです。

ここで感じたのは、「このデータ構造を理解していないと、BigQueryに繋いでも宝の持ち腐れになる」ということでした。

**気づき：UNNESTを理解することが、GA4×BigQuery活用の第一歩。**

## Day 4：セッションの概念が違う

4日目に取り組んだのは、セッション数のカウントです。GA4の標準レポートに表示されるセッション数と、BigQueryで集計したセッション数が合いません。

原因は、GA4のセッション定義がBigQueryのデータ上では明示的なカラムとして存在しないことでした。`ga_session_id` と `ga_session_number` は `event_params` の中にあり、これを組み合わせてセッションを識別する必要があります。

```sql
-- セッション単位の集計
SELECT
  (SELECT value.int_value FROM UNNEST(event_params) WHERE key = 'ga_session_id') AS session_id,
  user_pseudo_id,
  MIN(event_timestamp) AS session_start,
  MAX(event_timestamp) AS session_end
FROM `project.analytics_XXXXXXX.events_20260329`
GROUP BY session_id, user_pseudo_id
```

また、GA4の標準レポートとBigQueryの数値が完全一致しないことは、Googleの公式ドキュメントにも記載されています。サンプリングやデータ処理のタイミングの違いが原因です。

**気づき：GA4のレポートとBigQueryの数値は、完全一致を期待しない方が良い。**

## Day 5：データマートの必要性を痛感

5日目、毎回 `UNNEST` を含む長いSQLを書くのが苦痛になってきました。同じような集計を何度も書いており、非効率です。

ここで、「ビュー」を作成してデータを整理する必要性を痛感しました。いわゆるデータマートの設計です。

```
raw層（GA4生データ） → そのまま保持
  ↓
staging層（フラット化） → UNNESTを済ませた使いやすい形
  ↓
mart層（集計済み） → 日別売上、チャネル別セッション数など
```

この3層構造を最初から設計しておけば、日常的な分析では mart層のビューを `SELECT * FROM mart_traffic` のように呼ぶだけで済みます。

**気づき：データマート設計は「あとでやる」ではなく、最初にやるべき。**

## Day 6：最初のインサイトが見えた

6日目、staging層のビューを作成し、チャネル別のセッション数と売上を集計してみました。

すると、直接流入（Direct）のセッション数が全体の6割以上を占めていることがわかりました。これはGA4の標準レポートでも確認できる情報ですが、BigQueryで深掘りすると、Direct流入の中にもUTMパラメータが欠落しているSNS経由のトラフィックが含まれている可能性が見えてきました。

UTMパラメータの設定漏れを修正することで、チャネル別の分析精度が上がります。これは、BigQueryで生データを見なければ気づかなかったポイントでした。

**気づき：生データを直接見ることで、標準レポートでは見えない課題が見つかる。**

## Day 7：1週間の振り返り

1週間を終えて、感じたことをまとめます。

**良かったこと:**
- GA4の生データに直接アクセスできるようになった
- 標準レポートでは見えない粒度の分析ができるようになった
- データの蓄積が始まったことで、将来の分析の土台ができた

**苦労したこと:**
- UNNESTの書き方に慣れるまでの学習コスト
- セッションの定義を自分で組み立てる必要があること
- データマート設計の知識がないと、SQLが煩雑になること

**やっておけばよかったこと:**
- データマート設計を最初から行うこと
- UTMパラメータの運用ルールを先に整備すること
- 分析の目的（KPI）を明確にしてから導入すること

## これから導入する方へ

GA4×BigQueryの導入は、設定自体は簡単です。しかし、データを「使える状態」にするには、ある程度の学習コストと設計が必要です。

もし自社で一から取り組む時間が取れない場合は、初期のデータマート設計とビュー構築だけでも専門家に依頼すると、その後の運用がスムーズになります。

大事なのは、データの蓄積を早く始めることです。BigQueryに蓄積されたデータは、時間が経つほど価値が増します。
